# Java 天气 API 接入与封装落地实施文档

## 1. 项目背景与目标
在现代应用开发中，天气数据是提升用户体验的重要基础服务。本项目旨在通过接入第三方天气 API，构建一套稳定、安全且易于扩展的天气查询模块。
核心目标包括：
- 实现标准化的 HTTP 请求封装，解耦底层网络通信与上层业务逻辑。
- 建立完善的异常处理与防御性编程机制，确保服务的高可用性。
- 规范日志记录与数据解析流程，为后续的监控排查与功能迭代提供支撑。

## 2. 技术选型说明
基于 Java 生态的成熟工具链，本项目选用以下核心技术组件：
- **HTTP 客户端 (OkHttp)**：相比 JDK 原生的 `HttpURLConnection`，OkHttp 具备连接池复用、自动重试、透明 GZIP 压缩等优势，且 API 设计更为简洁。
- **JSON 解析 (Jackson)**：作为 Spring Boot 默认的序列化框架，Jackson 性能卓越，且支持通过注解灵活处理字段映射与未知属性。
- **日志门面 (SLF4J + Logback)**：遵循日志门面规范，支持占位符语法，避免字符串拼接带来的性能损耗，并便于统一配置日志级别与输出格式。
- **依赖管理 (Maven/Gradle)**：通过标准化的构建工具管理第三方依赖，确保版本的一致性与安全性。

## 3. 详细实现步骤

### 3.1 API 注册与凭证管理
- **服务商选择**：根据业务需求选择支持免费额度的天气服务商（如和风天气、OpenWeatherMap），并完成账号注册与应用创建。
- **安全配置**：严禁将 API Key 硬编码在代码中。应将其配置在 `application.yml` 或环境变量中，通过 `@ConfigurationProperties` 注入到配置类中。

### 3.2 HTTP 请求模块封装
- **超时设置**：构建 `OkHttpClient` 实例时，必须显式设置连接超时（Connect Timeout）与读取超时（Read Timeout），防止网络异常导致主线程长时间阻塞。
- **请求构建**：采用 Builder 模式构建 `Request` 对象，拼接基础域名、路径及查询参数。部分 API 要求必须设置 `User-Agent` 请求头，需按文档规范添加。
- **响应处理**：使用 try-with-resources 语法执行请求，确保 Response 资源被正确释放。优先校验 HTTP 状态码，再提取响应体字符串。

### 3.3 DTO 定义与 JSON 解析
- **数据模型映射**：定义与 API 返回 JSON 结构一致的 DTO 类。对于可能为空的数值型字段，务必使用 `Integer` 或 `Double` 等包装类，避免反序列化时抛出空指针异常。
- **容错配置**：配置 `ObjectMapper` 忽略未知属性（`FAIL_ON_UNKNOWN_PROPERTIES = false`），防止第三方 API 新增字段导致系统解析崩溃。
- **业务对象转换**：在 Service 层将外部 DTO 转换为内部业务对象，隔离外部接口结构变动对核心业务的影响。

### 3.4 异常处理机制
- **入参校验**：在发起网络请求前，校验城市名称是否为空或纯空白字符串。若校验失败，直接抛出明确的业务异常，避免无效的网络开销。
- **空值安全**：在处理嵌套的 JSON 数据时，推荐使用 Java 8 的 `Optional` 类进行链式调用，消除繁琐的 `if (obj != null)` 嵌套判断。
- **异常分类捕获**：分别捕获 `IOException`（网络层异常）和 `JsonProcessingException`（解析层异常），将其封装为统一的业务异常向上抛出，并记录完整的异常堆栈。

### 3.5 日志规范
- **日志级别使用**：正常业务流程使用 `INFO` 级别记录关键节点（如请求发起、响应状态码）；异常捕获使用 `ERROR` 级别，并传入异常对象以打印堆栈信息。
- **敏感信息脱敏**：在记录请求 URL 或参数时，必须对 API Key 进行脱敏处理，防止凭证泄露。
- **性能优化**：始终使用 SLF4J 的占位符语法（如 `logger.info("查询城市: {}", city)`）打印日志，避免在日志级别被禁用时产生不必要的字符串拼接开销。

## 4. 测试策略与用例
采用 JUnit 编写单元测试，确保核心逻辑的健壮性，测试用例需覆盖以下场景：
- **正常场景**：传入有效的知名城市名称，验证返回的业务对象字段是否完整且数据类型正确。
- **边界场景**：传入空字符串 `""`、纯空格 `"   "` 等非法入参，验证系统是否能快速失败并抛出预期的业务异常。
- **异常场景**：使用伪造的 API Key 或断网环境，验证 HTTP 异常捕获逻辑是否生效，以及错误提示是否友好。
- **Mock 测试**：使用 MockWebServer 或 Mockito 模拟第三方 API 的异常响应（如 401、404、429 状态码），验证系统的容错与降级逻辑。

## 5. 核心注意事项
- **API Key 安全**：API Key 是核心资产，除配置文件外，不得出现在代码、日志、Git 提交记录或前端暴露的接口中。
- **接口限流与重试**：第三方 API 通常有 QPS 限制。在封装层应引入令牌桶或滑动窗口限流机制，并在遇到 `429 Too Many Requests` 时实现带退避策略的重试机制。
- **连接池复用**：`OkHttpClient` 实例应作为全局单例使用，避免在每次请求时重新创建客户端导致连接池失效和资源泄露。
- **防御性编程**：永远不要信任外部 API 返回的数据结构。即使文档声明某字段必返回，在代码中也应做好空值判断与默认值兜底。
